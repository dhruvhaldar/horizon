## 2024-05-15 - [CRITICAL] DoS via Unbounded Mathematical Computation in Queueing Endpoint
**Vulnerability:** The `/api/queue` endpoint accepted unbounded arrays for server capacities `c`. The backend `mmc_queue` calculates exact factorials of `c` natively in Python `math.factorial()`. Submitting $c=100000$ triggers extremely expensive CPU computations and memory usage, hanging the worker process and causing Denial of Service.
**Learning:** Mathematical models require absolute bounds on parameters that grow exponentially or factorially. While the formulas are analytically correct, the underlying integer math implementation can be exploited if directly exposed to user input. The app did not have input validation layers (e.g., Pydantic constraints) restricting physical feasibility bounds on these variables.
**Prevention:** Implement strict numerical constraints directly on the API boundary (e.g. `c <= 100`). Always enforce physical limits when dealing with inputs mapping to combinatorial explosion or factorial calculations.

## 2024-05-16 - [MEDIUM] Information Leakage via Generic Exception Handling
**Vulnerability:** API endpoints were catching all exceptions using `except Exception as e` and directly returning the string representation `str(e)` in the HTTP response. This allows unhandled errors or internal exceptions to leak stack traces, implementation details, or sensitive server state directly to the user.
**Learning:** Returning generic exception strings can inadvertently expose sensitive data. When an unexpected error occurs, the user only needs to know there was an internal server error. Specific error details should only be returned for expected validation errors.
**Prevention:** Catch explicit and expected exceptions (e.g., `ValueError`) for returning user-facing HTTP 400 Bad Request messages. Let other unexpected exceptions bubble up or handle them generically so the framework can return a sanitized HTTP 500 response without leaking details.

## 2024-05-17 - [HIGH] Cross-Origin Resource Sharing (CORS) Vulnerability via Wildcard Origin and Credentials
**Vulnerability:** The FastAPI application used `CORSMiddleware` with `allow_origins=["*"]` while simultaneously setting `allow_credentials=True`. This allows arbitrary domains to bypass the Same-Origin Policy and potentially read sensitive responses using credentials (though none are currently used).
**Learning:** For stateless applications that don't rely on cookies or HTTP authentication, credentials are not needed. Setting `allow_credentials=True` when `allow_origins=["*"]` exposes the application to security risks. In fact, many modern browsers and frameworks outright block this exact combination, but FastAPI can sometimes dynamically reflect the Origin header.
**Prevention:** For stateless APIs, always ensure `CORSMiddleware` has `allow_credentials=False`. Only enable credentials when strictly necessary, and never pair it with wildcard `*` origins; explicitly whitelist allowed origins instead.
## 2024-05-18 - [CRITICAL] DoS via Unbounded Graph Algorithm Computation
**Vulnerability:** The `/api/route/tsp` and `/api/route/jobshop` endpoints accepted unbounded graphs (nodes, edges, jobs). The backend uses NetworkX algorithms (Floyd-Warshall, TSP approximation, Critical Path Method) whose time complexity scales non-linearly with graph size. Submitting a graph with 5,000 nodes triggers extremely expensive CPU computations and memory usage, hanging the worker process and causing Denial of Service.
**Learning:** Algorithmic complexity vulnerabilities occur when an application processes user input using algorithms that perform non-linearly (e.g., O(N^3) for Floyd-Warshall). Without explicit bounds, an attacker can submit a seemingly small request payload that expands into massive computational effort on the server.
**Prevention:** Implement strict numerical constraints directly on the API boundary. Validate that collections representing graph structures do not exceed safe sizes (e.g. `len(nodes) <= 100`, `len(edges) <= 500`).

## 2024-05-19 - [MEDIUM] Information Leakage via Unvalidated Graph Node Dependencies
**Vulnerability:** The `/api/route/jobshop` endpoint accepted job definitions that referenced non-existent dependencies (e.g., job A depends on job B, but job B is not defined). NetworkX automatically created nodes for these undefined dependencies without a 'duration' attribute. During the CPM calculation phase, referencing `G.nodes[pred]['duration']` raised an unhandled `KeyError` internal server error, which leaked the exception context.
**Learning:** Graph libraries like NetworkX often implicitly add nodes when edges refer to them. If algorithmic properties (like 'duration' or 'weight') are expected on all nodes, allowing implicitly created nodes will break the logic. Implicitly created state should not bypass input validation on explicitly defined nodes.
**Prevention:** Always validate graph dependencies against the canonical set of user-defined nodes at graph construction time. If a node is referenced that does not exist in the explicit definition map, immediately raise a client validation error (e.g., `ValueError`) to reject the payload and prevent downstream state inconsistencies.

## 2024-05-20 - [CRITICAL] DoS via Implicit Node Creation Bypassing API Limits
**Vulnerability:** The `/api/route/tsp` endpoint accepted a list of `nodes` and `edges`, limiting `len(nodes) <= 100` at the API boundary. However, NetworkX implicitly creates nodes when an edge references a non-existent node. An attacker could bypass the 100-node limit by sending an empty `nodes` list (passing validation) but including hundreds of edges referencing unique, undefined nodes. This constructs a massive graph implicitly, triggering non-linear algorithmic DoS (O(N^3) Floyd-Warshall).
**Learning:** API-level collection size limits (like `len(nodes) <= 100`) are useless if the underlying library (NetworkX) implicitly expands state based on unvalidated cross-references (like `edges`). The explicitly defined entities at the API boundary must be treated as a strict whitelist.
**Prevention:** Always perform canonical validation at graph construction time. Verify that all dependencies or edges exclusively reference the explicitly defined and validated set of nodes. Reject any payload containing implicitly created entities with a client validation error.

## 2024-05-21 - [MEDIUM] Information Leakage via Missing Array Validation leading to 500 Errors
**Vulnerability:** The `/api/queue` and `/api/route/tsp` endpoints lacked explicit validation ensuring array dimension consistency and list format requirements. **Prevention:** Always validate structured data interdependencies directly at the API boundary, or explicitly catch the resulting potential internal data-structure exceptions (like `IndexError` or `ValueError`) to return sanitized, user-facing 400 Bad Request errors.

## 2024-06-25 - [MEDIUM] Missing Probability Bounds Validation causing 500 JSON Serialization Errors
**Vulnerability:** The `/api/inventory/continuous` endpoint accepted a `service_level` probability value without bounding it between 0 and 1. When out-of-bounds values (like 1.5 or 0) were passed, `scipy.special.ndtri` returned `nan` or `-inf`. These non-finite float values are not JSON-compliant, causing the JSON encoder to crash with a `ValueError`, which propagated up as an unhandled 500 Internal Server Error, degrading stability and leaking internal error handling.
**Learning:** Statistical libraries (like SciPy) return non-finite values (like `nan` or `inf`) for invalid inputs to Probability Density Functions (PDF) or Cumulative Distribution Functions (CDF). These propagate up through internal calculations until they hit JSON serialization, which strictly forbids them, causing the application to crash generically.
**Prevention:** Always validate mathematical parameters that feed into statistical probabilities directly at the API layer. Ensure inputs intended to represent probabilities are strictly bounded (e.g., exclusively between 0 and 1).
