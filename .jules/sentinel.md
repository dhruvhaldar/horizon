## 2024-05-26 - Unhandled NaN/Infinity in Pydantic v2
**Vulnerability:** Pydantic v2 float fields accept "NaN" and "Infinity" string representations by default. This causes unhandled `ValueError` exceptions during FastAPI's `json.dumps()` serialization, exposing 500 Internal Server Errors (and potential stack traces if debug is on) and creating a potential Denial of Service vector.
**Learning:** This existed because Pydantic prioritizes loose coercion for standard types unless explicitly constrained, and the application did not validate that input floats were standard finite numbers.
**Prevention:** Always set `model_config = {"allow_inf_nan": False}` on Pydantic BaseModels handling numerical data from untrusted API requests to fail securely with a 422 Unprocessable Entity.
## 2024-05-26 - Unhandled NetworkXPointlessConcept Exception due to Duplicate TSP Nodes
**Vulnerability:** A vulnerability existed in the `/api/route/tsp` endpoint where duplicate nodes bypassing the `len(nodes) < 2` minimum check could be supplied. NetworkX treats graphs with duplicate elements mathematically as smaller graphs. When supplied `["A", "A"]`, a graph with < 2 elements was generated behind the scenes. Passing this to the TSP implementation triggered a `NetworkXPointlessConcept` Exception, bypassing our error checks and exposing a 500 Internal Server Error (and potentially stack traces or resource starvation DoS).
**Learning:** This existed because the application only checked the raw size of the requested array rather than the strict uniqueness of graph identifiers. Graph libraries usually deduplicate implicitly, changing the properties of the data dynamically compared to the original API assertions.
**Prevention:** Always validate graph representations by asserting that all nodes are strictly unique (e.g. `len(set(req.nodes)) == len(req.nodes)`) and meet mathematical bounds before passing to specialized libraries.
